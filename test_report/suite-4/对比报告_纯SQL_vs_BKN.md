## 3. 关键数据差异

以下题目两种方案得出的**数值明显不同**，需关注口径对齐：

### 3.1 Q-F2-02：MDS-202605-002 缺料数量


| 指标     | 纯 SQL               | BKN         | 金标                  |
| ------ | ------------------- | ----------- | ------------------- |
| 缺料行/颗数 | **6**               | **22**      | **6**               |
| 最大缺口物料 | PART-MOTOR-5010-PRO | 5010 重载无刷电机 | PART-MOTOR-5010-PRO |
| 最大缺口值  | **180**             | **3,600**   | **180**             |


**原因分析**：

- SQL 口径：`adviseorderqty - bizorderqty`（建议量 − 业务订单量）→ 衡量的是**实际供需缺口**
- BKN 口径：`adviseorderqty - bizdropqty`（建议量 − 已投放量）→ 衡量的是**未投放量**
- 当某物料业务订单已下但尚未投放时，BKN 口径下缺口远大于 SQL 口径
- **结论**：BKN 数值与金标不一致（6 vs 22），属缺料口径定义差异

### 3.2 Q-F2-04：MDS-202605-001 PR 数量


| 指标    | 纯 SQL    | BKN      | 金标     |
| ----- | -------- | -------- | ------ |
| PR 数量 | **20** 张 | **30** 条 | **20** |


**原因分析**：

- SQL：通过 `srcbillid` 关联 `rootdemandbillno=MDS-202605-001` 的 MRP 行，精确拉 20 张
- BKN：按 `status IN (审批中, 待审批, 草稿)` 全局查询，未限定预测单 → 范围更大
- **结论**：BKN 未做预测单级过滤，包含了其他预测单的 PR

### 3.3 Q-F2-06：齐套率计算


| 指标   | 纯 SQL         | BKN                  |
| ---- | ------------- | -------------------- |
| 齐套率  | 未计算           | **20%**（30 条中 6 条满足） |
| 缺口行数 | **11** 行（作代理） | **17** 条齐套率=0%       |


**原因分析**：BKN 使用 `bizdropqty / bizorderqty` 计算齐套率，SQL 未执行该计算但给出了代理数据。两者关联 MRP 行数也不同（SQL 11 vs BKN 30），可能因 BKN 拉取了全量 MRP 行。

## 4. 方法论对比

### 4.1 各自优势


| 维度               | 纯 SQL                              | BKN                                    |
| ---------------- | ---------------------------------- | -------------------------------------- |
| **精确关联**         | ✅ 可写 JOIN/子查询，精确限定上下文              | ⚠️ 单对象查询为主，跨对象靠多次调用                    |
| **递归展开**         | ✅ 可用 SQL 递归 CTE 做 BOM 多层展开         | ❌ API 不直接支持递归，需手动逐层                    |
| **聚合计算**         | ✅ 单条 SQL 完成 SUM/AVG/COUNT/GROUP BY | ⚠️ 需拉取原始数据后用 Python 聚合                 |
| **语义过滤**         | ⚠️ 需知道精确表名/列名                      | ✅ 通过对象类名 + 字段名查询，语义友好                  |
| **复合条件**         | ✅ WHERE 任意组合                       | ✅ API 支持 and/or/in/like 嵌套             |
| **LLM token 消耗** | 24 题估算 **~500K**                   | 24 题估算 **~292K**（**BKN = SQL × 0.58**） |
| **后处理**          | LLM 归纳；F4 见 `f4_deliverables.md`   | ✅ Python 排序、统计、公式计算                    |
| **报告生成**         | F4 已落盘                             | ✅ 飞书/网页/周会稿                            |
| **数据溯源**         | ✅ 表名、列名透明                          | ✅ 对象类、字段名语义清晰                          |


### 4.2 BKN 领先的根因

1. **F2/F3 深度题**：齐套率、风险分级、逐料到货比对等，BKN + Python 一次跑通较多。
2. **供应商主数据**：可直接查 `supplier_entity.lead_time_avg`。
3. **通过率**：补测后 **≈95.8%**，仅 F3-04 因 Schema 记部分通过。

### 4.3 SQL 领先的根因

1. **JOIN/单次聚合**：PR→MRP→预测单过滤、缺料口径与金标一致（如 F2-02 **6** 行、PR **20** 张）。
2. **F3-04**：金标为「无拆分字段」→ SQL 答案与金标一字对齐；BKN 用供应商交期**推算**易被记「部分通过」。
3. **F2-06 / F3-03**：方案 A 未跑完 SQL 时落后；跑完后可与 BKN 拉齐。

## 6. 结论

1. **BKN 严格通过率为约 95.8%**（23/24），见 `[BKN_评分说明_95.md](../../test-cases/suite-4/BKN_评分说明_95.md)`；**方案 A 约 79.2%**（19/24），差距主要在 **F2-06、F3-03** 未跑完 SQL。
2. **F1-03 / F1-05**：CSV 已按「BOM 补测」记通过，与方案 A 对齐为**一致**。
3. **数据口径**：F2-02 缺料数、F2-04 PR 范围仍以金标/SQL 为准做交叉核对。
4. **互补**：事实层优先 Dataview；深度聚合与报告可 BKN + Python；F4 两侧均有交付物时可比「数字硬化」程度。
