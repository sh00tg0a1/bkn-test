## 二、错误类型总览

| 错误类型 | 次数 | 占比 | 可规避 | 规避方案 |
|----------|------|------|--------|----------|
| json_parse（BOM 128KB 截断）| 4 | 44% | ✅ | limit≤30 分页 |
| invalid_condition_operator (`eq`) | 1 | 11% | ✅ | 只用 like/in |
| invalid_condition_operator (空条件) | 1 | 11% | ✅ | 使用 `--limit N` 无 body |
| internal_server_error (`match` 未构建) | 1 | 11% | ✅ | 执行 `bkn build` 或改用 like |
| internal_server_error (BOM 多条件) | 1 | 11% | ⚠️ 部分 | 先查全量再客户端过滤 |
| fetch failed（网络瞬时）| 1 | 11% | ✅ | 自动重试 |

**如果应用全部规避方案后的理论成功率**: 163 次调用中仅 1 次（BOM 多条件 500）难以规避 → **99.4%**

### 错误归因分析

9 次调用错误按根因归属到三个层面：

#### 后端服务问题（3 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| `eq` 操作符不支持 | HTTP 400 `InvalidParameter` | 1 | 文档列出但后端未实现 |
| BOM 多字段 like 触发 500 | HTTP 500 `GetViewDataByIDFailed` | 1 | `bom_material_code + bom_level` 双条件查询导致视图异常 |
| 空条件报错 | HTTP 400 `InvalidParameter` | 1 | `{"operation":"and","sub_conditions":[]}` 应允许或友好提示 |

#### SDK 问题（4 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| **stdout 128KB 硬截断** | CLI 输出超过 131072 bytes 时被静默截断，JSON 不完整 | **4** | exit code=0 但 JSON 不完整，AI/脚本无法通过返回码区分成功和截断，是**出现最多的错误** |

#### BKN 运维/操作遗漏（1 次）+ BKN 设计问题（非错误但影响准确性）

| 问题 | 现象 | 次数/影响 | 说明 |
|------|------|----------|------|
| `match` 未构建索引 | HTTP 500 | 1 次 | BKN 未执行 `kweaver bkn build`，`match` 无全文索引可查 |
| **历史数据未隔离** | PR/PO/工单返回全部历史批次 | **26 题** | BKN 数据视图未按预测批次预筛选，AI 必须自行区分 |
| BOM 多版本膨胀 | 103 种物料 × 9 版本 = 900 行 | 4 题 | 无内置最新版本过滤，是 128KB 截断的诱因 |
| supplier 对象类型无数据 | 查询返回 0 条 | 2 题 | 供应商数据未导入，只能从 PO 字段间接获取 |
| monitoring_task 仅测试数据 | 仅 1 条空记录 | 1 题 | 无法验证监控任务业务场景 |

#### 网络/基础设施（1 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| fetch failed | 网络瞬时错误 | 1 | 重试即可恢复 |

**操作符可用性总结**：
- `like`: 始终可用（基于 SQL 视图模糊匹配）
- `in` / `gt` / `lt` / `gte` / `lte`: 始终可用
- `match`: **需要 BKN 已执行 `build` 构建全文索引**，否则 500
- `eq`: 后端未实现，始终返回 400

## 四、Skill 优化建议（优先级排序）

### P0 必须修复（3项）

1. **BOM 查询 limit 上限设为 30**
   - 根因：CLI stdout 输出存在 128KB 硬截断
   - 影响：4 次 json_parse 错误（Q32/Q38/Q44）
   - 修复：`references/json-formats.md` 增加 "BOM 类大数据对象类型查询建议 limit≤30 并使用 search_after 分页"

2. **文档修正操作符支持列表**
   - 根因：`eq` 后端未实现（400）；`match` 需要 BKN 已 build 才可用（未 build 时 500）
   - 影响：2 次错误（Q01/Q09 首次尝试）
   - 修复：`references/json-formats.md` 中标注操作符可用性——`like`/`in`/`gt`/`lt`/`gte`/`lte` 始终可用；`match` 需已 build；`eq` 不可用

3. **增加"按预测批次过滤"标准查询模式**
   - 根因：历史数据陷阱
   - 修复方案示例：
     ```
     # 查当前批次的 PR → 通过 MRP 的 srcbillnumber 关联
     # 查当前批次的 PO → 通过 PR 的 billno 作为 srcbillnumber
     # 查当前批次的工单 → 通过工单的 planbegintime 或 srcbillnumber 过滤
     ```

### P1 建议修复（4项）

4. **增加自动重试机制**（fetch failed 瞬时错误）
5. **分页查询标准模板文档化**（search_after 用法）
6. **有效仓库过滤规则**（BKN 有效仓库列表应在查询示例中体现）
7. **空条件查询的正确用法**（使用 `--limit N` 无 body）

### P2 增强体验（7项）

8. count/stats API（避免分页计数）
9. supplier 对象类型数据补充
10. monitoring_task 数据完善
11. 多条件查询兼容性（BOM 视图 500）
12. BOM 按最新版本过滤
13. PR/PO → 预测单反向关联字段
14. Token 有效期延长或自动刷新

## 六、结论

1. **工具层面可靠性达标**：94.5% 成功率（规避已知问题后可达 99.4%），平均 4s 响应
2. **数据层面覆盖率高**：50 题中 100% 可通过工具查询获取所需数据（部分需分页/重试）
3. **AI 准确性最大挑战是历史数据区分**：PR/PO/工单/BOM 均存在历史版本混入风险
4. **3 个 P0 级修复项**可显著提升首次查询成功率和 AI 回答准确性
5. **BKN 数据质量良好**：核心业务数据（MRP/PO/库存/预测单/工单）均能通过 API 获取并支撑业务分析
