## 二、错误类型总览

| 错误类型 | 次数 | 占比 | 可规避 | 规避方案 |
|----------|------|------|--------|----------|
| json_parse（BOM 128KB 截断）| 4 | 44% | ✅ | limit≤30 分页 |
| invalid_condition_operator (`eq`) | 1 | 11% | ✅ | 只用 like/in |
| invalid_condition_operator (空条件) | 1 | 11% | ✅ | 使用 `--limit N` 无 body |
| internal_server_error (`match` 未构建) | 1 | 11% | ✅ | 执行 `bkn build` 或改用 like |
| internal_server_error (BOM 多条件) | 1 | 11% | ⚠️ 部分 | 先查全量再客户端过滤 |
| fetch failed（网络瞬时）| 1 | 11% | ✅ | 自动重试 |

**如果应用全部规避方案后的理论成功率**: 162 次调用中仅 1 次（BOM 多条件 500）难以规避 → **99.4%**

### 错误归因分析

9 次调用错误按根因归属到三个层面：

#### 后端服务问题（3 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| `eq` 操作符不支持 | HTTP 400 `InvalidParameter` | 1 | 文档列出但后端未实现 |
| BOM 多字段 like 触发 500 | HTTP 500 `GetViewDataByIDFailed` | 1 | `bom_material_code + bom_level` 双条件查询导致视图异常 |
| 空条件报错 | HTTP 400 `InvalidParameter` | 1 | `{"operation":"and","sub_conditions":[]}` 应允许或友好提示 |

#### SDK 问题（4 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| **stdout 128KB 硬截断** | CLI 输出超过 131072 bytes 时被静默截断，JSON 不完整 | **4** | exit code=0 但 JSON 不完整，AI/脚本无法通过返回码区分成功和截断 |

#### BKN 运维/操作遗漏（1 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| `match` 未构建索引 | HTTP 500 | 1 | BKN 未执行 `kweaver bkn build`，`match` 无全文索引可查 |

#### 网络/基础设施（1 次）

| 问题 | 现象 | 次数 | 说明 |
|------|------|------|------|
| fetch failed | 网络瞬时错误 | 1 | 重试即可恢复 |

#### 数据层面问题（不影响错误率，但影响 AI 回答准确性）

| 问题 | 影响范围 | 说明 |
|------|----------|------|
| **历史数据未隔离** | ~26 题 | PR/PO/工单返回全部历史批次，AI 必须自行区分当前批次 |
| BOM 多版本膨胀 | 4 题 | 无内置最新版本过滤，是 128KB 截断的诱因 |
| supplier 对象类型无数据 | 2 题 | 供应商数据未导入，只能从 PO 字段间接获取 |
| monitoring_task 仅测试数据 | 1 题 | 仅 1 条空记录，无法验证监控任务业务场景 |

#### 影响优先级

```
影响范围（高→低）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

──────────┼────────────────────────────┤
SDK       │████████████████            │  4次  128KB 静默截断（出现最多的错误）
──────────┼────────────────────────────┤
后端服务   │██████████                  │  3次  操作符不支持 / 视图兼容性
──────────┼────────────────────────────┤
```

**操作符可用性总结**：
- `like`: 始终可用（基于 SQL 视图模糊匹配）
- `in` / `gt` / `lt` / `gte` / `lte`: 始终可用
- `match`: **需要 BKN 已执行 `build` 构建全文索引**，否则 500
- `eq`: 后端未实现，始终返回 400

## 四、AI 回答能力评估

### 50 题回答能力总结

| 评级 | 题数 | 占比 | 含义 |
|------|------|------|------|
| ✅ 可直接回答 | 36 | 72% | 数据充分，无需额外处理 |
| ⚠️ 可回答但需注意 | 10 | 20% | 数据有但存在历史陷阱或数据变化 |
| ❌→✅ 需补查后回答 | 4 | 8% | 首次查询失败，分页/重试后可回答 |

### 最大风险：历史数据陷阱

由于未使用 SKILL.md 中的批次过滤规则，**历史数据混入问题更加突出**：

| 对象类型 | 陷阱描述 | 影响题目数 |
|----------|----------|-----------|
| PR/PO | 按物料编码查返回所有历史批次，不仅限当前预测 | 12 题 |
| 工单(MPS) | T01 有12条历史工单、948 有5条（或更多），大部分已完工 | 6 题 |
| 预测单 | T01 有4张预测单，AI 需按 billno 定位 | 4 题 |
| BOM | T01 有9个版本，数据量膨胀 9 倍 | 4 题 |

**与优化版 BKN 对比**：优化版通过 SKILL.md 中的批次过滤规则（如通过 MRP 的 srcbillnumber 关联 PR、通过 PR 的 billno 作为 srcbillnumber 关联 PO）可以在一定程度上隔离当前批次数据。未优化版缺少这些规则，AI 更容易将历史 PR/PO 与当前批次混淆。

## 六、Skill 优化建议（优先级排序）

### P0 必须修复（3项）

1. **BOM 查询 limit 上限设为 30**
   - 根因：CLI stdout 输出存在 128KB 硬截断
   - 影响：4 次 json_parse 错误（Q32/Q38/Q44）
   - 修复：文档增加 "BOM 类大数据对象类型查询建议 limit≤30 并使用 search_after 分页"

2. **文档修正操作符支持列表**
   - 根因：`eq` 后端未实现（400）；`match` 需要 BKN 已 build 才可用（未 build 时 500）
   - 影响：2 次错误（Q01/Q09 首次尝试）
   - 修复：标注操作符可用性——`like`/`in`/`gt`/`lt`/`gte`/`lte` 始终可用；`match` 需已 build；`eq` 不可用

3. **增加"按预测批次过滤"标准查询模式**
   - 根因：历史数据陷阱（**影响 26 题，是准确性最大障碍**）
   - 修复方案示例：
     ```
     # 查当前批次的 PR → 通过 MRP 的 srcbillnumber 关联
     # 查当前批次的 PO → 通过 PR 的 billno 作为 srcbillnumber
     # 查当前批次的工单 → 通过工单的 planbegintime 或 srcbillnumber 过滤
     ```

### P1 建议修复（4项）

4. **增加自动重试机制**（fetch failed 瞬时错误）
5. **分页查询标准模板文档化**（search_after 用法）
6. **有效仓库过滤规则**（如昆山成品仓为有效仓、委外加工仓和借用仓需确认）
7. **空条件查询的正确用法**（使用 `--limit N` 无 body）

### P2 增强体验（7项）

8. count/stats API（避免分页计数）
9. supplier 对象类型数据补充
10. monitoring_task 数据完善
11. 多条件查询兼容性（BOM 视图 500）
12. BOM 按最新版本过滤
13. PR/PO → 预测单反向关联字段
14. Token 有效期延长或自动刷新

## 八、结论

1. **工具层面可靠性达标**：94.4% 成功率（规避已知问题后可达 99.4%），平均 3.9s 响应
2. **与优化版 BKN 表现一致**：两个 BKN 错误率、错误模式、响应时间几乎完全相同，说明问题根源在 SDK/后端/数据设计层面，而非 BKN 定义差异
3. **数据层面覆盖率高**：50 题中 100% 可通过工具查询获取所需数据（部分需分页/重试）
4. **AI 准确性最大挑战是历史数据区分**：PR/PO/工单/BOM 均存在历史版本混入风险，SKILL.md 的批次过滤规则能缓解但无法根治
5. **3 个 P0 级修复项**可显著提升首次查询成功率和 AI 回答准确性
6. **BKN 数据质量良好**：核心业务数据（MRP/PO/库存/预测单/工单）均能通过 API 获取并支撑业务分析
