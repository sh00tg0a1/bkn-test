## 四、错误归因与改进建议

### 4.1 SQL 口径误解（S2-Q10）

**问题**：题目「深圳天翼下面有几个厂」指的是集团旗下全部工厂，而非工厂名称中含「深圳天翼」的子集。

**改进**：对口语题中的企业/集团名称，应先查询 `factory_entity` 全部记录（或按集团标识字段过滤），再统计，而不是用集团名称做 `LIKE` 过滤工厂名称字段。

---

### 4.2 计数字段选择错误（S2-Q29）

**问题**：题目问「采购申请单总共有多少条」，应按主键 `pr_id` 或表行数计（106），被测用 `COUNT(DISTINCT billno)` 得 105，因为存在 1 个 billno 对应多个 entry 行的情形。

**改进**：区分「单据数」（`DISTINCT billno`）和「行数」（主键 `pr_id`）两个口径；题目问「多少条」时，应按主键行数计，除非题目明确指"多少张单"。

---

### 4.3 上文仓库 ID 解析错误（S3-Q01）

**问题**：S1-Q02 的金标注释指出，「各仓合计 1215」由 WH001（−2155）+ WH002（+3370）构成，金标将 WH002（电子材料仓）称为「惠州仓」。但 S3-Q01 要求「只看惠州仓」时，被测用 `WH004` 或名称含「惠州」匹配，漏查了正确的 WH002。

**改进**：追问类题目应**追溯上文假设中的数据口径**，将上文出现的仓库与实际 warehouse_id 对应关系建立上下文；若不确定，应先查 `warehouse_entity` 全表确认业务名称与 ID 的对应。

---

### 4.4 边界回应策略不足（S4-Q17）

**问题**：金标期望回答「可先提供数据结论要点，完整报告/邮件发送需配置报告服务」，即提供部分帮助 + 说明限制。被测直接拒绝，未给出积极的替代方案。

**改进**：对「超出接口能力」的请求，应采用「**能做什么 + 做不到什么 + 建议路径**」的三段式回应，而不是直接拒绝。例如：「我可以先把关键数据整理给你（…），完整网页/邮件报告需要配置报告服务，建议联系系统管理员。」

## 五、结论

本次测试（`test_run_20260326221800`）通过直接数据库查询验证了 Suite-3 全部 100 题，**整体正确率 96%**，在以下方向存在少量错漏：

1. **口语题的查询口径**：集团名称≠工厂名称关键词，需识别真实业务含义再构造 SQL。
2. **主键 vs 业务单据号的计数区分**：「多少条」vs「多少张」需要明确字段选择策略。
3. **追问上下文中的实体解析**：仓库、物料等实体应从上文假设中继承具体 ID，而非独立重新模糊匹配。
4. **边界题的回应策略**：对「无法执行」类请求应给出积极引导，而非单纯拒绝。
