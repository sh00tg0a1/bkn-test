### C.3 按题型分析

- **S1（1 跳，100%）**：单表直查无误，数据一致性良好。
- **S2（2 跳，76%）**：4 题因 warehouse_code 编码体系不一致（WH-华中-01 vs WH001）导致语义虽近但编码不匹配；建议统一仓库编码映射。
- **S3（3 跳，60%）**：多跳 JOIN 复杂度上升，FK 链路断裂（customer_code↔customer_id）和 SQL 未精确限定范围是主要问题。
- **S4（4 跳，72%）**：4~5 表 JOIN 时部分 FK 路径不通（物料采购→供应商→发货），SQL 未限定目标产品导致范围溢出。

### C.4 改进建议

1. **仓库编码统一**：确认 `warehouse_code`（WH001 vs WH-华中-01）在数据源中的实际值，SQL 按实际编码过滤。
2. **FK 关联路径**：`customer_code` 与 `customer_id` 可能需要先子查询映射，再用于 JOIN。
3. **SQL 范围控制**：多跳查询须在 JOIN 条件中精确限定目标产品/客户，避免笛卡尔积扩散。
4. **聚合字段**：涉及汇总金额/库存的题目应在 SQL 中加 SUM/GROUP BY，避免依赖多行 JSON 的人工加总。
5. **负值排查**：出现负 `inv_qty` 说明 LEFT JOIN 与 GROUP BY 的交互异常，需调整查询逻辑。
